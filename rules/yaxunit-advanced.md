# Расширенные паттерны YaXUnit

Применяется: при написании юнит-тестов через фреймворк YaXUnit для 1С.

Базовые паттерны тестирования (шаблон теста, типы кейсов, цели покрытия) — см. `testing-patterns.md`.

---

## Именование тестовых модулей

Тестовые модули — общие модули в расширении или каталоге `tests/src/CommonModules`.

### Префиксы (по типу тестируемого объекта)

| Префикс | Тип объекта | Пример |
|---------|------------|--------|
| ОМ_ | Общий модуль | ОМ_РаботаСФайлами_Тесты |
| Док_ | Документ | Док_ЗаказКлиента_Тесты |
| Спр_ | Справочник | Спр_Номенклатура_Тесты |
| РН_ | Регистр накопления | РН_ТоварыНаСкладах_Тесты |
| РС_ | Регистр сведений | РС_КурсыВалют_Тесты |
| РБ_ | Регистр бухгалтерии | РБ_Хозрасчетный_Тесты |
| Обр_ | Обработка | Обр_ЗагрузкаДанных_Тесты |
| Отч_ | Отчёт | Отч_АнализПродаж_Тесты |

### Суффиксы (по типу модуля объекта)

| Суффикс | Тестируемый модуль | Пример |
|---------|-------------------|--------|
| (без суффикса) | Общий модуль / основной | Док_ЗаказКлиента_Тесты |
| _МО | Модуль объекта | Док_ЗаказКлиента_МО_Тесты |
| _ММ | Модуль менеджера | Спр_Номенклатура_ММ_Тесты |
| _НЗ | Набор записей | РН_ТоварыНаСкладах_НЗ_Тесты |
| _МФ | Модуль формы | Док_ЗаказКлиента_МФ_Тесты |

## Структура тестового модуля

```bsl
#Область СлужебныйПрограммныйИнтерфейс

Процедура ИсполняемыеСценарии() Экспорт

    ЮТТесты
        .ДобавитьТест("ТестЗаписьНовогоЭлемента")
        .ДобавитьТест("ТестЗаписьСНевернымиДанными")
        .ДобавитьТест("ТестПроведениеДокумента")
            .СПараметрами("Товар1", 10)
            .СПараметрами("Товар2", 0);

КонецПроцедуры

#КонецОбласти

#Область Обработчики

Процедура ПередВсемиТестами() Экспорт
    // Подготовка общих данных для всех тестов модуля
КонецПроцедуры

Процедура ПослеВсехТестов() Экспорт
    // Очистка после всех тестов
КонецПроцедуры

#КонецОбласти

#Область Тесты

Процедура ТестЗаписьНовогоЭлемента() Экспорт
    // Arrange / Act / Assert
КонецПроцедуры

#КонецОбласти
```

Параметризованные тесты через `.СПараметрами()` — фреймворк вызывает тест отдельно для каждого набора параметров.

## API ассертов

Основной метод: `ЮТест.ОжидаетЧто(Значение)` — возвращает fluent-объект для цепочки проверок.

### Сравнение

| Метод | Описание |
|-------|----------|
| `.Равно(Ожидаемое)` | Строгое равенство |
| `.НеРавно(Значение)` | Неравенство |
| `.Больше(Значение)` | Больше |
| `.БольшеИлиРавно(Значение)` | Больше или равно |
| `.Меньше(Значение)` | Меньше |
| `.МеньшеИлиРавно(Значение)` | Меньше или равно |

### Типы и заполненность

| Метод | Описание |
|-------|----------|
| `.ЭтоНеопределено()` | Значение = Неопределено |
| `.ЭтоНеНеопределено()` | Значение <> Неопределено |
| `.ЭтоИстина()` | Значение = Истина |
| `.ЭтоЛожь()` | Значение = Ложь |
| `.ЭтоNull()` | Значение = Null |
| `.ИмеетТип(Тип)` | Проверка типа |
| `.Заполнено()` | ЗначениеЗаполнено() = Истина |
| `.НеЗаполнено()` | ЗначениеЗаполнено() = Ложь |
| `.Существует()` | Ссылка не помечена на удаление |

### Коллекции

| Метод | Описание |
|-------|----------|
| `.ИмеетДлину(N)` | Количество элементов = N |
| `.ИмеетДлинуБольше(N)` | Количество > N |
| `.Содержит(Элемент)` | Коллекция содержит элемент |
| `.НеСодержит(Элемент)` | Коллекция не содержит элемент |
| `.ВСписке(Список)` | Значение входит в список |

### Строки

| Метод | Описание |
|-------|----------|
| `.НачинаетсяС(Префикс)` | Строка начинается с |
| `.ЗаканчиваетсяНа(Суффикс)` | Строка заканчивается на |
| `.СодержитСтроку(Подстрока)` | Строка содержит подстроку |
| `.СодержитСтрокуПоШаблону(Шаблон)` | Соответствие регулярному выражению |

### Свойства объектов

| Метод | Описание |
|-------|----------|
| `.Свойство(Имя)` | Переключиться на проверку свойства |
| `.ИмеетСвойство(Имя)` | Свойство существует |
| `.НеИмеетСвойства(Имя)` | Свойство не существует |
| `.ИмеетСвойстваРавные(Структура)` | Несколько свойств равны ожидаемым |

### Пример fluent-цепочки

```bsl
ЮТест.ОжидаетЧто(Результат)
    .ИмеетТип("Структура")
    .Свойство("Наименование").Равно("Тест")
    .Свойство("Количество").Больше(0)
    .Свойство("Дата").Заполнено();
```

## Генерация тестовых данных

### КонструкторОбъекта

```bsl
// Правильно: явные значения, воспроизводимый тест
Конструктор = ЮТест.Данные().КонструкторОбъекта("Справочники.Номенклатура")
    .Установить("Наименование", "Тестовый товар")
    .Установить("Артикул", "АРТ-001")
    .ФикцияОбязательныхПолей()
    .Записать();

СсылкаНоменклатуры = Конструктор.Ссылка();

// Неправильно: все поля случайные — тест нестабилен
Конструктор = ЮТест.Данные().КонструкторОбъекта("Справочники.Номенклатура")
    .Фикция()
    .Записать();
```

Методы КонструкторОбъекта:
- `.Установить(Имя, Значение)` — задать конкретное значение реквизита
- `.Фикция(Имя)` — заполнить реквизит случайным значением подходящего типа
- `.ФикцияОбязательныхПолей()` — заполнить все обязательные поля случайными значениями
- `.ТабличнаяЧасть(Имя)` — начать заполнение табличной части
- `.ДобавитьСтроку()` — добавить строку в табличную часть
- `.Записать()` — записать объект в БД
- `.Провести()` — записать и провести (для документов)
- `.Ссылка()` — получить ссылку на записанный объект

Правило: использовать `.Фикция()` только для несущественных полей. Значения, важные для теста — задавать явно через `.Установить()`.

## Проверки базы данных

```bsl
// Проверка что в регистре появилась запись
ЮТест.ОжидаетЧтоТаблицаБазы("РегистрНакопления.ТоварыНаСкладах")
    .Содержит(
        ЮТест.Предикат()
            .Получить("Номенклатура").Равно(СсылкаНоменклатуры)
            .Получить("Склад").Равно(СсылкаСклада)
            .Получить("КоличествоПриход").Больше(0)
    );
```

`ЮТест.Предикат()` — fluent-построитель условий фильтрации записей в таблице БД.

## DESIGNER vs EDT

| Аспект | DESIGNER (XML) | EDT (.mdo) |
|--------|---------------|------------|
| Каталог тестов | `tests/src/CommonModules/` | `src/CommonModules/` в проекте расширения |
| Формат модуля | `.xml` с BSL внутри | `.bsl` файлы напрямую |
| Флаг серверного модуля | В XML: `<Server>true</Server>` | В `.mdo`: `server="true"` |
| Запуск | vanessa-runner или 1cv8 ENTERPRISE | EDT Test Runner |

Для доступа к базе данных из тестов модуль должен быть серверным (`server=true`).
