# Архитектурное ревью 1С

Применяется: перед реализацией значительных архитектурных изменений в конфигурациях 1С.

---

## Когда проводить

- Перед добавлением новых подсистем
- При изменении ключевых структур данных (регистры, справочники-владельцы)
- При проектировании интеграций с внешними системами
- При введении новых архитектурных паттернов
- После крупного рефакторинга

---

## Что ревьюировать

### Дизайн метаданных

| Аспект | Что проверить |
|--------|---------------|
| Справочники | Подходят для мастер-данных? Иерархия обоснована? |
| Документы | Корректная схема движений? Операции правильно отражены? |
| Регистры накопления | Правильные измерения/ресурсы? Производительность при больших объёмах? |
| Регистры сведений | Подходящая периодичность? Не дублируют справочники? |
| Общие модули | Чёткое разделение ответственности? Корректная область видимости экспорта? |

### Структура модулей

- Принцип единой ответственности (SRP) соблюдён
- Минимальная связанность между модулями
- Общая логика вынесена в общие модули
- Код пригоден для тестирования

### Клиент-серверное взаимодействие

- `&НаСервереБезКонтекста` используется где возможно
- Количество клиент-серверных вызовов минимизировано
- Передаются только необходимые данные
- Длительные операции выполняются асинхронно

### Доступ к данным

- Пакетные запросы вместо запросов в цикле
- Получение реквизитов через БСП, а не через точку
- Стратегия кеширования обоснована
- Границы транзакций определены корректно

### Безопасность

- RLS спроектирован корректно
- Привилегированный режим используется минимально и обоснованно
- Пользовательский ввод валидируется
- Аудит критических операций обеспечен

## Red flags

Немедленно отмечать как CRITICAL:

- **God-модуль** — один модуль содержит сотни процедур без разделения
- **Нарушение клиент-серверной границы** — серверная логика на клиенте или обратно
- **Прямой SQL** — использование внешних источников данных без обоснования
- **Жёсткие ссылки** — привязка к конкретным UUID или предопределённым элементам без проверки
- **Обход RLS** — вызов `УстановитьПривилегированныйРежим(Истина)` без документированного обоснования
- **Отсутствие обработки ошибок** — `Попытка/Исключение` с пустым блоком `Исключение`
