# Безопасная запись BSL через write_module_source

Самая опасная зона при работе с кодом 1С через EDT MCP: неверный режим записи может затереть весь модуль или потерять часть кода. Прочитать ДО первой записи в модуль.

Применяется: при записи BSL через `write_module_source` (1c-edt). Дополняет `file-edit-efficiency.md` (выбор инструмента) и `mcp-tool-priority.md` (раздел записи BSL).

---

## Режимы write_module_source и их риск

| Режим | Что делает | Риск |
|-------|-----------|------|
| `search_replace` | Находит фрагмент и заменяет. Затрагивает только найденное | Низкий - режим по умолчанию для точечной правки |
| `replace_method` | Заменяет один метод целиком по имени | Низкий - безопасен для правки метода |
| `replace_lines` | Заменяет диапазон строк (с защитой `expectedText`) | Средний - см. ниже |
| `insert_before` / `insert_after` | Вставка относительно якоря | Низкий |
| `append` | Добавляет в конец модуля | Низкий |
| `replace` | **Перезаписывает ВЕСЬ файл модуля** | Высокий - см. правило 1 |

После записи EDT автоматически валидирует результат (синтаксис) и возвращает ошибки в ответе - проверять их сразу.

---

## Правило 1: НИКОГДА не использовать replace для одного метода

`replace` заменяет весь файл модуля. Передав один метод - удалишь весь остальной код (сотни/тысячи строк), без отката.

Для правки одного метода - `replace_method` (по имени) или `search_replace` (старый текст метода в поиск, новый в замену). `replace` оправдан только когда: модуль создается впервые, либо модуль пустой/менее ~50 строк и передается его полное содержимое, либо полная перезапись модуля по явному согласованию с пользователем.

## Правило 2: oldSource (search_replace) = доказательство, что прочитал текущий код

Для `search_replace` искомый фрагмент должен встречаться в файле РОВНО один раз. Ноль совпадений - операция отклонена; больше одного - тоже отклонена с подсказкой взять более крупный уникальный фрагмент. Это защищает от замены не того места и подтверждает, что ты видел актуальное содержимое. При ошибке "not found / multiple" - НЕ ретраить вслепую: перечитать метод (`read_method_source`) и обновить/расширить искомый фрагмент до уникального.

## Правило 3: replace_lines требует expectedText

`replace_lines` сверяет фактический текст заменяемых строк с переданным `expectedText`. Если код в файле сдвинулся (номера строк устарели) - запись отклоняется с diff, файл не меняется. Всегда передавать `expectedText` и не доверять номерам строк из старого чтения.

---

## Алгоритм безопасной правки метода

1. `read_method_source` - прочитать метод, запомнить текущее состояние (и номера строк).
2. При необходимости оценить размер модуля - `get_module_structure`.
3. Сформировать `write_module_source`: `mode=replace_method` (или `search_replace` с уникальным `oldSource`), новый полный текст метода (с директивами `&НаКлиенте`/`&НаСервере`/`&НаСервереБезКонтекста`).
4. Проверить ответ на синтаксические ошибки (EDT валидирует автоматически).
5. `get_project_errors` с фильтром по затронутому объекту - убедиться, что не появилось семантических ошибок.

## Алгоритм отката неудачной записи

Встроенного undo у `write_module_source` нет. Откат держится на том, что ты помнишь предыдущее содержимое ДО записи.

1. **Перед каждой `search_replace` держать в контексте и `oldSource` (что заменяешь), и новый текст** - не выбрасывать, пока не подтвердил корректность записи.
2. Если в ответе видно, что заменен не тот фрагмент или сломалась структура - **немедленно** выполнить обратную `search_replace`: передать новый текст как `oldSource`, а прежний `oldSource` как замену. Чем быстрее, тем меньше шанс, что открытый редактор EDT увидит сломанный файл.
3. Если обратная замена уже невозможна (новый текст не уникален в файле, или модуль прошел через `replace`) - **остановиться и спросить пользователя**. НЕ реконструировать модуль "по памяти": в EDT может быть открыта копия в редакторе, восстановление через UI надежнее.
4. Для `replace` правило то же: держать полный старый текст до отправки нового, иначе откат невозможен.

---

## Чего нельзя делать

- Править `.bsl` напрямую через `Edit`/`Write`/`Read+Write` - только через `write_module_source`, иначе EDT не увидит изменений до refresh, и теряется авто-проверка и подсчет строк (см. `mcp-tool-priority.md`: Serena/прямое редактирование отключены для BSL).
- `replace` без полного содержимого нового модуля.
- Передавать усеченное содержимое в надежде "сервер разберется" - запишется ровно то, что передано.
- `skipSyntaxCheck=true` чтобы "обойти" ошибку баланса блоков - это почти всегда симптом реального бага в коде.
- Коммитить, если `get_project_errors` показывает новые проблемы в измененном модуле.
