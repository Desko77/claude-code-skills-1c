# Работа с БСП (Библиотека стандартных подсистем)

Правила работы в конфигурациях на БСП: переиспользование готового, разметка правок типовых модулей, устаревшие объекты.

Применяется: при доработке любой конфигурации на БСП (ERP, УТ, БП, ЗУП, КА и самописные на БСП). Если проект НЕ на БСП - правило неактуально.

Дополняет `1c-coding-standards.md` (общий стиль BSL) и `mcp-tool-priority.md` (инструмент `search_ssl`).

---

## Принцип: сначала БСП, потом свой код

Перед тем как писать свою реализацию - проверить, нет ли готового решения в БСП. Своя реализация поверх готового модуля БСП - технический долг.

### Порядок для новой функциональности

1. Поиск в БСП - `search_ssl` (1c-mcp_ssl_server) по описанию задачи; либо `get_metadata_objects metadataType=commonModules nameFilter=<тема>` (1c-edt) по тематическому ключу (Файлы, ЭлектроннаяПочта, Печать, Пользователи, ...).
2. Изучить методы найденного модуля - `get_module_structure`, `read_method_source` для интересующих.
3. Если решение есть - использовать его.
4. Только если решения нет - писать свое. И в комментарии указать, почему БСП не подошла.

### Ключевые модули БСП

| Модуль | Назначение |
|--------|-----------|
| `ОбщегоНазначения` / `ОбщегоНазначенияКлиентСервер` | Общие утилиты |
| `Пользователи` | Пользователи, роли, права |
| `РаботаСФайлами` | Файлы и присоединенные объекты |
| `УправлениеПечатью` | Печатные формы |
| `ДлительныеОперации` | Фоновые задания с прогрессом |
| `ВерсионированиеОбъектов` | История версий объектов |
| `РаботаСПочтовымиСообщениями` | Отправка почты |
| `СтроковыеФункцииКлиентСервер` | Строковые утилиты |

Частые утилиты: `ОбщегоНазначения.ЗначениеРеквизитаОбъекта(Ссылка, "Реквизит")` (без загрузки всего объекта), `ОбщегоНазначения.МенеджерОбъектаПоСсылке(Ссылка)`, `ОбщегоНазначения.ПодсистемаСуществует(Имя)` (перед условным вызовом), `ОбщегоНазначенияКлиентСервер.СвойствоСтруктуры(Стр, "Ключ", Умолч)`.

### Суффиксы модулей

| Суффикс | Среда | Кеш |
|---------|-------|-----|
| (без суффикса) | Сервер | Нет |
| `Клиент` | Клиент | Нет |
| `КлиентСервер` | Обе | Нет |
| `ПовтИсп` | Сервер | На сеанс |

В клиентском коде сначала искать `*КлиентСервер`, затем `*Клиент`. В серверном - базовый модуль или `*Сервер`. `*ПовтИсп` - кеш на сеанс, для часто запрашиваемых справочных данных.

## Разметка правок типовых модулей (маркеры модификации)

Любое изменение типового модуля БСП оборачивать в маркеры. Это нужно, чтобы при обновлении БСП было видно, какие места правились, и перенести правки.

### Формат маркеров

- Открывающий: `// <<МОДИФИКАЦИЯ` (без пробела между `<<` и словом). Опционально - короткая пометка после пробела или ` - `: `// <<МОДИФИКАЦИЯ - Расширен список адресатов`.
- Закрывающий: `// >>`.
- Для однострочной правки (изменена сигнатура и т.п.) допустим только хвостовой маркер: `Процедура ИмяПроцедуры(...) Экспорт // <<МОДИФИКАЦИЯ` - без `// >>`.
- Маркеры - это merge-метаданные, не журнал. Не писать в них номера задач, даты, авторов - для этого есть git.

Префикс зависит от проекта/команды: где-то `<<МОДИФИКАЦИЯ`, где-то `<<КОМПАНИЯ`, `<<ОРГ` и т.п. **Сверять фактический префикс по уже размеченным правкам в конкретной конфигурации** и использовать его же; не навязывать свой.

### Сохранение оригинальной строки

При замене или удалении строки типового кода - оставлять оригинал закомментированным внутри блока. При обновлении БСП будет видно, что и на что заменено.

```bsl
// <<МОДИФИКАЦИЯ
//	ШаблоныЗаданий.Добавить(Метаданные.РегламентныеЗадания.ПолучениеИОтправкаЭлектронныхПисем.Имя);
ШаблоныЗаданий.Добавить("ПолучениеИОтправкаЭлектронныхПисем");
// >>
```

Для чистого добавления (типовая строка не заменяется) - просто обернуть новый код:

```bsl
// <<МОДИФИКАЦИЯ - Заполнение списка документов
СписокДокументов.Добавить("#Задание", НСтр("ru = 'Задание'"));
// >>
```

Для чистого удаления - удаленная строка остается закомментированной внутри маркеров, без замены.

### Когда применять

- Любая правка `.bsl` типового модуля (`СтандартныеПодсистемы.*`, `ОбщегоНазначения`, `Пользователи`, `РаботаСФайлами` и т.п.).
- Правки внешних библиотек, обновляемых извне - та же конвенция.
- Свои модули проекта - маркеры НЕ нужны: это свой код, обновление БСП его не затронет.
- В расширениях (заимствование) разметка обычно не нужна - механизм расширений сам изолирует правку; маркеры - для прямых правок типового кода в основной конфигурации.

## Префикс `Удалить` - устаревшие объекты

В конфигурациях на БСП объекты метаданных и реквизиты с префиксом `Удалить` (`УдалитьКонтрагент`, `УдалитьАдрес` и т.п.) - устаревшие заглушки, оставленные для обратной совместимости. Считать их несуществующими: не читать, не писать, не ссылаться, не расширять в новом коде. Для замены искать актуальный объект без префикса.
