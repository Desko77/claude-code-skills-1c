# Workflow оптимизации производительности 1С

Применяется: при подозрении на проблемы производительности, после code review, перед деплоем в продакшен.

Список антипаттернов — см. `anti_patterns.md`.
Советы по оптимизации запросов — см. `query-optimization-tips.md`.

---

## Когда применять

- Пользователь сообщает о медленной работе
- Code review выявил потенциальные проблемы производительности
- Перед деплоем нового функционала с обработкой больших объёмов данных
- Регулярный аудит производительности

Не применять для преждевременной оптимизации — сначала измерить.

---

## Workflow анализа

### Шаг 1. Поиск hot spots

Искать антипаттерны из `anti_patterns.md` в коде:

| Что искать | Паттерн поиска | Серьёзность |
|------------|----------------|-------------|
| Запрос в цикле | `Для Каждого` + `Новый Запрос` | CRITICAL |
| Реквизиты через точку | `.Реквизит` у ссылочных типов | CRITICAL |
| Подзапрос в SELECT | Вложенный `ВЫБРАТЬ` в списке полей | CRITICAL |
| Фильтр ВТ в ГДЕ | Условия на виртуальных таблицах | HIGH |
| Нет ПЕРВЫЕ N | Запросы без ограничения | HIGH |
| Множественные серверные вызовы | Последовательные `НаСервере` с клиента | HIGH |
| &НаСервере вместо БезКонтекста | Серверный вызов без использования контекста | HIGH |
| Нет кеширования | Повторные вызовы с одинаковыми параметрами | MEDIUM |
| O(n^2) | Вложенные циклы | MEDIUM |

Инструменты: `search_in_code` (1c-edt), Grep по .bsl-файлам.

### Шаг 2. Оценка приоритета

```
Приоритет = Влияние * Частота * Объём данных

CRITICAL: исправить немедленно
  - Запрос в цикле на больших данных
  - Реквизиты через точку в цикле
  - Подзапросы с большими выборками

HIGH: исправить до деплоя
  - Фильтры виртуальных таблиц
  - Избыточные серверные вызовы
  - Отсутствие ПЕРВЫЕ N на больших таблицах

MEDIUM: исправить в спринте
  - Отсутствие кеширования
  - Неоптимальные алгоритмы
```

### Шаг 3. Профилирование (при необходимости)

Если нужны точные замеры — использовать технологический журнал:
- Скил `1c-techlog` — настройка logcfg.xml, захват событий DBMSSQL/SDBL с фильтром duration
- Анализ длительности запросов и частоты вызовов

### Шаг 4. Оптимизация

Для каждого найденного антипаттерна:
1. Проверить текущее поведение (если есть тесты — запустить)
2. Применить минимальное изменение — решение из `anti_patterns.md` и `query-optimization-tips.md`
3. Проверить что функциональность сохранена
4. Задокументировать улучшение

### Шаг 5. Верификация

- Повторный поиск антипаттернов — убедиться что все исправлены
- Запуск тестов — функциональность не сломана
- При наличии технологического журнала — сравнить замеры до/после

---

## Контрольные вопросы

Перед завершением оптимизации проверить:

- [ ] Все CRITICAL антипаттерны устранены?
- [ ] Запросы в циклах заменены на пакетные?
- [ ] Обращения через точку заменены на методы БСП?
- [ ] Виртуальные таблицы фильтруются через параметры, а не ГДЕ?
- [ ] Серверные вызовы минимизированы?
- [ ] Функциональность не нарушена?
