# validate_api_reference.py - линтер API-справочников

Проверяет reference-файлы скилов (например, `skills/zup-hr-api-reference/references/`)
против реальной выгрузки конфигурации 1С: каждый упомянутый в бэктиках токен
`Модуль.Метод` должен существовать в выгрузке как экспортный метод общего модуля.
Защита от галлюцинаций: опечатки и выдуманные имена модулей/методов в справочнике
всплывают до того, как их скопирует агент.

Адаптация валидатора из [brake71/1c-ssl-skills](https://github.com/brake71/1c-ssl-skills)
(MIT, (c) Чекменев Дмитрий Алексеевич).

## Синтаксис

```
python tools/validate_api_reference.py --refs <файл-или-папка-md> --src <выгрузка> [--json]
python tools/validate_api_reference.py --refs <файл-или-папка-md> --list [--json]
```

- `--src` - корень выгрузки конфигурации, папка с `CommonModules/`. Поддержаны
  выгрузка Конфигуратора (`<Модуль>/Ext/Module.bsl`, UTF-8 с BOM) и EDT-проект
  (`<Модуль>/Module.bsl`; можно указать корень проекта - подпапка `src/` найдется сама).
- `--list` - без выгрузки: только извлечь токены и показать, что принято и что
  отфильтровано (визуальная проверка экстрактора).
- `--json` - машиночитаемый отчет.

Коды выхода: 0 - чисто, 1 - есть находки (ERROR/WARN), 2 - ошибка параметров.

## Пример (справочник ЗУП)

```
python tools/validate_api_reference.py --refs "skills/zup-hr-api-reference/references" --src "C:/dumps/zup31"
python tools/validate_api_reference.py --refs "skills/zup-hr-api-reference/references" --list
```

## Что ловит

- модуль не найден в выгрузке (опечатка или выдуманный общий модуль) - ERROR;
- метод не найден в модуле - ERROR;
- метод найден, но не экспортный - ERROR;
- рядом с токеном заявлено "стабильный", а метод лежит в служебном/устаревшем
  регионе `#Область` - WARN (вне отслеживаемых областей - ERROR);
- заявлено "служебный", а метод в `ПрограммныйИнтерфейс` - INFO.

## Что извлекается и что фильтруется

Берутся только токены в бэктиках: inline-спаны с полным совпадением
(`Модуль.Метод`, `Модуль.Метод()`) и вызовы `Модуль.Метод(` внутри fenced-блоков
кода. Оба сегмента - идентификаторы с заглавной буквы (кириллица/латиница).

Фильтруются (не проверяются): типы метаданных (`Документ.X`, `РегистрСведений.X`
и англ. аналоги), менеджеры объектов (`Документы.X...`, `РегистрыСведений.X...`),
цепочки из 3+ сегментов, имена файлов (`*.md`, `*.py`), вызовы методов
платформенных объектов на переменных (`ЗапросВТ.Выполнить()` - см. стоп-лист
`PLATFORM_OBJECT_METHODS`), а также токен сразу после маркера "НЕТ" - конвенция
разделов "частые ошибки" ("НЕТ метода `X.Y()` - использовать `X.Z()`": первый
токен намеренно несуществующий и пропускается, замена проверяется).

## Ограничения

- Только общие модули: методы менеджеров, модулей объектов и форм не проверяются.
- Нужна локальная выгрузка конфигурации нужной версии; результат осмыслен ровно
  для той версии/поставки (КОРП-методы не найдутся в выгрузке ПРОФ).
- Inline-спан с путем данных вида `Структура.Ключ` (оба слова с заглавной буквы,
  не модуль) даст ложный ERROR "модуль не найден" - перепроверить руками и
  переформулировать справочник (убрать бэктики или разбить на два спана).
- Заявление стабильности ищется в тексте той же строки (слова "стабильный" /
  "служебный"); в коде - только в комментарии после `//`.
