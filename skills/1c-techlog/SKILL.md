# 1c-techlog — Технологический журнал 1С

## Когда использовать

- Диагностика ошибок платформы (EXCP)
- Анализ блокировок и взаимоблокировок (TLOCK, TDEADLOCK)
- Профилирование медленных запросов (DBMSSQL, SDBL)
- Мониторинг соединений и процессов (CONN, PROC)
- Настройка logcfg.xml по запросу пользователя

---

## Правила безопасности

1. **Сохранить оригинал** — перед изменением logcfg.xml прочитать и сохранить текущее содержимое (или запомнить что файла не было)
2. **Восстановить после работы** — после анализа вернуть оригинальный logcfg.xml или удалить если его не было
3. **Не оставлять надолго** — техжурнал непрерывно пишет на диск; оставленный на часы/дни может заполнить диск
4. **Проверить свободное место** — перед включением убедиться что на целевом диске достаточно места
5. **Сообщить пользователю** — после завершения анализа явно подтвердить что техжурнал восстановлен/отключён

---

## Workflow

```
1. СОХРАНИТЬ оригинальный logcfg.xml (прочитать или запомнить отсутствие)
2. СОЗДАТЬ logcfg.xml с нужными событиями (выбрать шаблон ниже)
3. ДОЖДАТЬСЯ события (пользователь воспроизводит проблему или запускает тест)
4. ОТКЛЮЧИТЬ техжурнал (удалить logcfg.xml или записать пустую конфигурацию)
5. ПОДОЖДАТЬ обработку логов парсером (~10-30 секунд)
6. СОБРАТЬ логи — прочитать файлы из каталога логов
7. АНАЛИЗ — найти релевантные события, определить причину
8. ВОССТАНОВИТЬ оригинальный logcfg.xml
9. ПРОВЕРИТЬ — подтвердить пользователю что техжурнал в исходном состоянии
```

## Выбор шаблона

| Шаблон | Когда использовать | Примерный объём |
|--------|-------------------|-----------------|
| Минимальный (EXCP) | Диагностика исключений | ~1-10 МБ/сутки |
| Блокировки (TLOCK) | Анализ блокировок и deadlock | ~10-100 МБ/сутки |
| Производительность (DBMSSQL) | Медленные запросы | ~100 МБ - 1 ГБ/сутки |
| Комплексный | Полная диагностика | ~1-10 ГБ/сутки |

Подробные XML-шаблоны — см. `references/templates.md`.

## Структура logcfg.xml

```xml
<?xml version="1.0" encoding="UTF-8"?>
<config xmlns="http://v8.1c.ru/v8/tech-log">
    <dump create="false"/>
    <log location="D:\1CLogs\DiagName" history="24">
        <event>
            <eq property="name" value="excp"/>
        </event>
        <event>
            <eq property="name" value="dbmssql"/>
            <gt property="duration" value="5000000"/>
        </event>
        <property name="all"/>
    </log>
</config>
```

Ключевые элементы:
- `<log location="...">` — каталог для логов (использовать отдельный диск, не системный)
- `history="N"` — хранить логи за N часов
- `<event>` — фильтр по типу события
- `<eq property="name" value="..."/>` — имя события
- `<gt property="duration" value="N"/>` — фильтр по длительности (микросекунды)
- `<property name="all"/>` — записывать все свойства события

## Типичные ошибки

- Логи на системном диске (C:\) — может заполнить и остановить сервер
- Нет фильтра `duration` для DBMSSQL — логируются ВСЕ запросы, огромный объём
- Забыли отключить после диагностики — диск заполняется за часы/дни
- `history` слишком большой — хранение логов за недели занимает десятки ГБ
- Нет `<dump create="false"/>` — создаются дампы аварийного завершения

## Примеры использования

### Диагностика взаимоблокировки

1. Настроить logcfg.xml с шаблоном "Блокировки" (см. `references/templates.md`)
2. Попросить пользователя воспроизвести ситуацию
3. В логах искать события TDEADLOCK — они содержат:
   - `WaitConnections` — соединения, участвующие в deadlock
   - `DeadlockConnectionIntersections` — пересечения блокировок
4. По `connectID` из TDEADLOCK найти соответствующие TLOCK-события
5. Определить какие объекты блокировались и из каких модулей

### Анализ медленных запросов

1. Настроить logcfg.xml с шаблоном "Производительность", `duration > 3000000` (3 сек)
2. Попросить пользователя выполнить проблемную операцию
3. В логах DBMSSQL/SDBL найти запросы с наибольшим `duration`
4. Проанализировать текст запроса (`Sql` или `Sdbl` свойства)
5. Рекомендовать оптимизацию — см. `rules/query-optimization-tips.md`
